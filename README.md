# SDN-FIREWALL

Problem Statement

Traditional network security relies on static, hardware-bound middleboxes that are difficult to reconfigure and lack centralized visibility. This project demonstrates the implementation of a Software-Defined Networking (SDN) firewall. By leveraging the separation of the control plane and data plane, a centralized POX controller is programmed to manage traffic flows dynamically. The objective is to enforce security policies that permit or deny communication between specific hosts within a Mininet topology through explicit OpenFlow rules and real-time packet inspection.

Setup and Execution Steps
1. Prerequisites
The environment requires a Linux-based system (Ubuntu) with the following dependencies installed:

Mininet Network Emulator

POX SDN Controller framework

Wireshark Network Analyzer

Python 3.x

2. File Placement and Preparation
The project architecture consists of two primary scripts. Proper placement is required for execution:

topology.py: Save this file in the primary workspace directory.

firewall.py: Save this file in the pox/ext directory to enable the POX controller to load it as a module.

3. Running the Simulation
Execute the project using two separate terminal instances to observe the interaction between the controller and the switch.

Terminal 1: Launching the POX Controller

Bash
cd ~/pox
python3 pox.py firewall
Terminal 2: Launching the Mininet Topology

Bash
sudo mn --custom topology.py --topo orangetopo --controller remote
Expected Output and Functional Validation
1. Rule-Based Filtering Verification
The firewall implements a specific security policy: Block traffic from Host 1 (10.0.0.1) to Host 2 (10.0.0.2) while permitting all other communication.

Test Scenario (Blocked): Executing h1 ping h2 results in 100% packet loss.

Test Scenario (Allowed): Executing h1 ping h3 results in 0% packet loss with successful Echo Replies.

Network-wide Audit: The pingall command confirms that only the path between h1 and h2 is dropped, while all other connections remain functional.

2. Deep Packet Inspection via Wireshark
Wireshark is the primary tool for monitoring the OpenFlow protocol exchange and verifying the Match-Action logic. By filtering for openflow_10 and icmp, the following sequence is observed:

OFPT_PACKET_IN: This message is generated by the switch when h1 attempts a new connection. The switch sends the packet header to the controller because no matching flow entry exists.

OFPT_PACKET_OUT: This message is generated by the controller to instruct the switch on how to handle the packet.

Firewall Logic: For blocked traffic (h1 to h2), the controller suppresses the packet-out command and does not install a forwarding flow, effectively dropping the traffic. For allowed traffic (h1 to h3), the controller issues the packet-out command to the correct port.

3. Controller Logging and Auditing
The POX controller terminal provides a real-time audit trail for security events. When a violation of the security policy occurs, the following log is generated:
INFO: --- FIREWALL: Dropping packet from 10.0.0.1 to 10.0.0.2 ---
